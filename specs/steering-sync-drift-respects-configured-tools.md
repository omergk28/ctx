# Steering Sync Drift Respects Configured Tools

*Origin: session 2263caef (2026-06-07), during a `/ctx-drift` pass. A
`ctx steering sync --all` run surfaced that `ctx drift` nags every
project to sync steering for cursor/cline/kiro regardless of which
tools it actually uses. Scoped as a ctx behavior change; supersedes the
hardcoded tool trio in `internal/drift/check_ext.go:checkSyncStaleness`.*

## Problem

`ctx drift`'s sync-staleness check is hardwired to three tools and
treats a **missing** synced output as stale. In `internal/drift/check_ext.go`
the check iterates a literal `syncTools := []string{ToolCursor, ToolCline,
ToolKiro}` and calls `steering.StaleFiles` for each; `StaleFiles` →
`unchanged(outPath, content)` returns false when `outPath` does not
exist, so absence is reported as drift.

The consequence: a project that uses only Claude — which reads
`.context/steering/` directly and needs no synced files — is told it has
12 stale steering files (3 tools × 4 steering files) it can only silence
by generating tool-native outputs for editors it never opens. This repo
hit exactly that. Drift's value is its signal-to-noise ratio; a warning
that fires for tools you don't use is pure noise that trains the reader
(human or agent) to ignore the whole report.

**Why now:** the noise is live in this repo today, and the same defect
ships to every ctx end-user. The fix is small and self-contained.

The tension the fix must hold: the ctx **dev team is multi-tool**, so
the *repo* legitimately wants cursor/cline/kiro outputs tracked and kept
fresh — while an **end-user on a single tool** must not be nagged about
the others. Both are served by one mechanism (below), not by picking a
side.

## Approach

Replace the hardcoded trio with a **presence-based** tools-in-play set:

```
toolsInPlay = { t ∈ syncableTools : steering.Synced(projectRoot, t) }
```

A syncable tool is "in play" iff its native output already exists on
disk. Having synced once (and, for a repo, committing the result) is
itself the opt-in; drift then keeps it honest. `syncableTools` is
`cursor`, `cline`, `kiro` only — `claude` and `codex` consume steering
via `ctx agent` directly and are never file-synced.

Why presence alone is sufficient — it covers every case the earlier
"explicit declaration" idea was invented for:

- **Multi-tool repo "ships all"** — it *tracks* the cursor/cline/kiro
  outputs, so they are present in every clone → drift enforces them. No
  declaration field needed.
- **Single-tool end-user** — never synced the others → no outputs
  present → no nag. No config needed.
- **Repo treating outputs as build artifacts** (gitignored, not
  committed) — drift correctly stays quiet; freshness is instead
  guaranteed by the `make check-steering` gate below.

Empty set → the check passes silently. No new outputs are generated by
drift (it stays read-only); generation remains `ctx steering sync`'s job.

Second, close the maintenance gap: wire `sync-steering` +
`check-steering` into `make build`, mirroring the existing
`sync-copilot-skills` / `check-copilot-skills` pattern. This is the
**sole** mechanism for repo-side freshness — authoritative whether or
not the outputs are tracked, and independent of drift's per-project
presence signal.

**Deliberately no new config field.** An earlier draft proposed a
`.ctxrc` `tools:` list to *declare* targets. Presence-detection plus
`make check-steering` make it unnecessary, and it would collide
semantically with the existing `steering.default_tools` (see Non-Goals).
Dropped.

**Where it fits:** `checkSyncStaleness` (drift) changes its tool-set
source; `internal/steering` gains a presence helper and exposes its
syncable set; the Makefile gains two targets. No `.ctxrc` change, no
change to native formats or to how steering is consumed.

## Behavior

### Happy Path

1. `ctx drift` runs `checkSyncStaleness`.
2. It computes `toolsInPlay = { t ∈ syncableTools : steering.Synced(projectRoot, t) }` (presence only).
3. For each tool in the set it calls `steering.StaleFiles(...)` and emits
   one warning per stale (or missing) file, tagged with the tool.
4. If the set is empty, the check records `Passed` and emits nothing.

### Edge Cases

| Case | Expected behavior |
|------|-------------------|
| Claude-only project: no synced outputs present | Set empty → check passes, **zero** warnings (the headline fix). |
| Tool synced once, output present + fresh | In play → checked, clean. |
| Tool synced, then steering source edited | In play → its now-stale file(s) reported, for that tool only. |
| Output present but partially stale | Only the stale files listed, per current `StaleFiles` behavior. |
| Output dir exists but empty (e.g. all files tombstoned) | Not "in play" if no native file present (see Open Questions on granularity); no spurious warning. |
| Steering source dir absent or empty | Check passes (nothing to sync) — unchanged from today. |
| Tombstoned steering file | Skipped (existing `HasTombstone` behavior), no output expected, not reported. |
| Repo tracks outputs; fresh clone | Outputs present → enforced from first `ctx drift`, no per-user setup. |

### Validation Rules

- No new config to validate. The set is derived purely from filesystem
  presence under `projectRoot`, intersected with the syncable tools.
- `claude`/`codex` are never in the set (not syncable), so they neither
  warn nor error here regardless of any `.ctxrc` `tool` value.
- Presence detection is read-only filesystem state; it never causes a
  write.

### Error Handling

| Error condition | User-facing message | Recovery |
|-----------------|---------------------|----------|
| Stale/missing output for an in-play tool | existing `DescKeyDriftStaleSyncFile` + `<name> (<tool>)` suffix | `ctx steering sync --tool <tool>` (or `--all`) |
| `rc.ContextDir()` fails | existing context-dir error surfaced as a drift warning | fix `.ctxrc` / run from a ctx project (unchanged) |
| `make check-steering` finds stale tracked outputs | `FAIL: steering outputs are stale — run 'make sync-steering'` (mirroring the copilot-skills check) | run `make sync-steering`, commit the regenerated outputs |

## Interface

### CLI

No new command. Existing `ctx steering sync [--tool X | --all]` is
unchanged. `ctx drift` output simply stops listing tools that aren't in
play.

### Configuration (`.ctxrc`)

**None.** No new `.ctxrc` key. The check is driven entirely by on-disk
presence. (The pre-existing, unused `steering.default_tools` is left
alone — see Non-Goals.)

### Make targets

```
make sync-steering    # ctx steering sync --all (regenerate tracked outputs)
make check-steering   # non-zero if any tracked output is stale
```

`build:` gains `sync-steering`; the aggregate `check` gains
`check-steering`.

### Key Functions (sketch — finalize at implementation)

```go
// checkSyncStaleness (new shape)
func checkSyncStaleness(report *Report) {
    steeringDir := rc.SteeringDir()
    files, err := steering.LoadAll(steeringDir)
    if err != nil || len(files) == 0 {
        report.Passed = append(report.Passed, cfgDrift.CheckSyncStaleness)
        return
    }
    projectRoot := /* filepath.Dir(rc.ContextDir()) — unchanged */
    found := false
    for _, tool := range steering.SyncableTools() {
        if !steering.Synced(projectRoot, tool) {
            continue // not in play: never synced for this project
        }
        for _, name := range steering.StaleFiles(steeringDir, projectRoot, tool) {
            report.Warnings = append(report.Warnings, /* same Issue as today */)
            found = true
        }
    }
    if !found {
        report.Passed = append(report.Passed, cfgDrift.CheckSyncStaleness)
    }
}
```

`steering.Synced(projectRoot, tool)` reports whether the tool's native
output already exists (granularity per Open Questions).

## Testing

- **Unit (`steering.Synced`):** true when a tool's native output exists,
  false otherwise; never writes. Cover the empty-dir / all-tombstoned
  granularity case.
- **Integration (drift):** fixtures —
  (a) Claude-only, no synced outputs → `checkSyncStaleness` Passed, zero
  warnings (the headline fix);
  (b) cursor synced + fresh → clean, and cline/kiro absent → not checked;
  (c) cursor synced then source edited → stale warning for cursor only,
  none for cline/kiro;
  (d) all three synced + fresh (the multi-tool-repo clone) → clean,
  all three enforced.
- **Make:** `check-steering` exits non-zero after a deliberately-staled
  tracked output and zero after `sync-steering`.
- **Regression:** the existing `internal/drift/check_ext_test.go`
  staleness cases updated to the presence model (the prior cases assumed
  the hardcoded trio is always checked).

## Non-Goals

- Not changing how `claude`/`codex` consume steering (`ctx agent`
  direct read) — they remain out of the sync/staleness path.
- Drift stays **read-only**: it never auto-syncs or writes outputs;
  generation is `ctx steering sync`'s job.
- Not adding support for new sync target tools, nor changing any
  tool's native output format.
- **No new `.ctxrc` config.** An explicit declared-targets field was
  considered and dropped — presence-detection + `make check-steering`
  cover its cases without a new surface.
- **`steering.default_tools` is out of scope.** It is a separate,
  pre-existing, fully unused field (`internal/rc/types.go`) whose
  *documented* purpose is the default `Tools:` frontmatter for **new
  steering files** — a content-scoping concern distinct from this
  output-scoping check. Its cleanup (wire it to that purpose, or remove
  it from struct + schema + docs) belongs in its own change. This spec
  must not conflate the two.
- The *feature* does not require tracking the generated steering
  outputs — drift is correct via presence either way, and freshness is
  gated by `make check-steering`. (Operational note: **this repo opts to
  track** cursor/cline/kiro outputs, decided 2026-06-07, so a fresh
  clone has them present and drift enforces them for the multi-tool dev
  team. That's a repo policy, not a spec requirement.)

## Open Questions

- **Presence granularity.** Detect a tool as present by output-dir
  existence, or by at least one matching native file? Dir-exists is
  simpler; file-level avoids a stale/empty dir counting as "in play".
  Lean: file-level (matches how `StaleFiles` already enumerates per
  file). Settle at implementation.
